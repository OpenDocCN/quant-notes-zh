## 课程反馈与作业回顾

上一节我们介绍了市场微观结构和tick数据的基本概念。本节中，我们首先来看看大家第一次作业的完成情况，并针对一些共性问题进行说明。

### 作业反馈要点

以下是针对第一次作业，需要大家注意的几个核心点：

1.  **集合竞价处理**：在期货交易中，开盘前（例如08:55-08:59）会进行集合竞价。此时市场挂单但不成交，交易所根据最大成交量原则确定开盘价。在数据处理时，通常需要过滤掉开盘前（如09:00前）的集合竞价数据。
2.  **盘中集合竞价**：某些品种（如期权）在价格波动剧烈时，会触发盘中集合竞价，旨在让市场冷静。此时可能出现连续时间成交量为零，但盘口价格变动的情况，这并非数据错误。
3.  **夜盘数据处理**：对于夜盘结束时间（如23:00, 23:30, 01:00），一个常见的处理技巧是：最后一个tick的时间通常不会超过收盘时间点后500毫秒。可以使用一个微小的时间阈值（如小于收盘时间后1秒）来保留或过滤最后一个tick，这取决于个人研究习惯。
4.  **涨跌停处理**：在涨跌停时，盘口数据可能呈现特殊状态（如涨停时卖一量为零）。在合成低频K线时，需要特别注意这种情况，避免误判为数据错误。判断标准是：买一和卖一价格同时为零通常意味着数据错误；而只有一边为零可能是正常的涨跌停状态。

## 进阶数据Bar类型介绍

在掌握了基础的Tick和分钟Bar之后，我们可以探索一些更高级的Bar构建方法，这对于机器学习建模可能更有益处。

上一节我们处理了按时间切分的Bar。本节中我们来看看按成交量或成交额切分的Bar。

传统的Tick Bar或分钟Bar没有考虑每笔交易的量能信息。有人认为，成交100手的一个Tick和分10个Tick成交、每笔10手，所包含的市场信息是不同的。因此，可以尝试以下两种Bar：

1.  **Volume Bar（成交量Bar）**：不按时间，而是按累计成交量切分。例如，每成交10万手就生成一个Bar。
    *   **动机**：使每个Bar承载的交易活动量相对均衡。
2.  **Dollar Bar（成交额Bar）**：按累计成交额切分。例如，每成交1000万元就生成一个Bar。
    *   **动机**：同时考虑了成交量和价格，更能反映市场实际流入的资金量。

**核心概念对比**：
*   **Tick/Time Bar**: `Bar_count = f(timestamp_interval)`
*   **Volume Bar**: `Bar_count = f(cumulative_volume)`
*   **Dollar Bar**: `Bar_count = f(cumulative_volume * price)`

有研究表明，Volume Bar和Dollar Bar的收益率序列，可能比Time Bar更接近独立同分布（I.I.D.），这有利于许多统计模型和机器学习模型的假设。鼓励有兴趣的同学将此作为拓展练习，验证不同Bar类型下收益率序列的统计特性。

**注意**：使用Volume Bar或Dollar Bar时，在开盘、收盘边界需要特殊处理（例如，收盘时累计量未达到阈值，是截断还是延续至次日开盘），这没有标准答案，取决于具体的研究目标和品种特性。

## CTP交易系统简介

前面我们学习了数据的形态和处理。本节中我们来看看产生这些数据的源头——交易系统。

CTP（Comprehensive Transaction Platform）是上海期货交易所推出的综合交易平台。一个完整的程序化交易流程通常涉及以下三部分：

1.  **交易系统（Trading System）**：这是我们策略程序直接交互的部分。它负责接收行情（Market Data），并根据策略逻辑发出订单（Order）、管理订单（如撤单Cancel）。
2.  **风控管理（Risk Management）**：通常由期货公司（券商）执行。功能包括：
    *   **验资**：检查账户资金是否足够下单。
    *   **保证金监控**：监控账户风险度，在保证金不足时通知追加或强行平仓。
3.  **结算（Settlement）**：由交易所完成。每日收盘后，交易所会向客户发送结算单，详细列出当日所有成交记录、持仓、资金变动等。交易员需要做“对账（Reconcile）”，确保自己的记录与结算单一致。

### 交易系统通信模式

CTP API采用客户端-服务器模式，有三种通信方式：

1.  **请求/应答模式（Request/Response）**：由客户端主动发起请求（如查询合约信息），服务器应答。数据流：`Client -> 期货公司柜台 -> 交易所 -> 期货公司柜台 -> Client`。
2.  **私有流模式（Private Mode）**：由交易所主动向特定客户端推送私有信息（如订单成交回报、资金变动）。这类信息是客户私有的。
3.  **广播模式（Broadcast）**：交易所向所有订阅的客户端广播公共信息（即**行情数据Market Data**）。为了保证公平，交易所会尽力让所有客户端同时收到行情。

**降低延迟**：对于高频交易，降低每一步的通信延迟至关重要。常见方法包括：将服务器托管在交易所机房（缩短物理距离）、选择技术更优的期货公司柜台、优化网络硬件和协议。

### API结构解析

![](img/cf7c4c79136b818b283308f5417c9034_90.png)

CTP API主要包含两个核心接口：
*   **MdApi**：用于行情订阅和接收。
*   **TraderApi**：用于交易操作（下单、撤单、查询等）。

今天我们重点学习`MdApi`。其工作模式是典型的“继承与回调”：

![](img/cf7c4c79136b818b283308f5417c9034_92.png)

![](img/cf7c4c79136b818b283308f5417c9034_94.png)

1.  **继承SPI类**：我们需要创建一个自己的类（例如`CMyMdSpi`），继承自CTP提供的`CThostFtdcMdSpi`类。这个SPI（Strategy Programming Interface）类定义了一系列虚函数（回调函数）。
2.  **实现回调函数**：在我们的类中，实现这些虚函数。当特定事件发生时（如连接建立、行情到达），CTP底层会自动调用对应的函数。
3.  **创建API实例并注册SPI**：创建`CThostFtdcMdApi`实例，并将我们的`CMyMdSpi`实例注册给它。
4.  **初始化连接**：调用API的初始化、连接前置服务器、登录等方法。
5.  **订阅行情**：登录成功后，订阅指定的合约代码。
6.  **接收行情**：行情数据到达时，我们实现的`OnRtnDepthMarketData`函数会被调用，在这里处理收到的tick数据。

```cpp
// 1. 创建自定义SPI类，继承自CThostFtdcMdSpi
class CMyMdSpi : public CThostFtdcMdSpi {
public:
    // 构造函数，传入API指针
    CMyMdSpi(CThostFtdcMdApi* pApi) : m_pApi(pApi) {}

    // 2. 实现关键的回调函数
    /// 当连接前置机成功时被调用
    virtual void OnFrontConnected() {
        // 执行登录操作
        CThostFtdcReqUserLoginField req;
        // ... 填充账号、密码等信息
        m_pApi->ReqUserLogin(&req, ++m_requestId);
    }

    /// 登录响应
    virtual void OnRspUserLogin(...) {
        // 登录成功，开始订阅行情
        char* ppInstruments[] = {"rb2010"}; // 订阅螺纹钢2010合约
        m_pApi->SubscribeMarketData(ppInstruments, 1);
    }

    /// 3. 最重要的回调：深度行情通知
    virtual void OnRtnDepthMarketData(CThostFtdcDepthMarketDataField *pDepthMarketData) {
        // 在这里处理收到的tick数据！
        // pDepthMarketData 结构体中包含最新价、买卖一价、成交量等所有信息
        // 可以将其打印、或写入文件/数据库
        saveToCsv(pDepthMarketData);
    }

    // ... 实现其他必要的回调函数，如错误处理、心跳等
private:
    CThostFtdcMdApi* m_pApi;
    int m_requestId = 0;
};

// 主程序
int main() {
    // 4. 创建MdApi实例
    CThostFtdcMdApi* pMdApi = CThostFtdcMdApi::CreateFtdcMdApi();
    // 5. 创建我们的SPI实例，并注册
    CMyMdSpi mySpi(pMdApi);
    pMdApi->RegisterSpi(&mySpi);
    // 6. 注册前置机地址并初始化
    pMdApi->RegisterFront("tcp://simnow-front:12345"); // SimNow前置地址
    pMdApi->Init();
    // 7. 等待回调事件发生（通常使用死循环或事件驱动）
    pMdApi->Join();
    return 0;
}
```

**关键数据结构**：`CThostFtdcDepthMarketDataField` 是包含深度行情数据的结构体。你需要查阅CTP头文件（如`ThostFtdcUserApiStruct.h`），了解其具体字段（如`LastPrice`, `BidPrice1`, `AskPrice1`, `Volume`等），这与我们之前处理的tick数据字段是对应的。

### 1. 猜数字游戏（博弈论）

**问题**：我在1-100之间随机选一个数N。你猜一个数K。如果你猜对了（K=N），我给你K元；否则，你给我1元。你应该采取什么策略？

![](img/cf7c4c79136b818b283308f5417c9034_121.png)

![](img/cf7c4c79136b818b283308f5417c9034_123.png)

![](img/cf7c4c79136b818b283308f5417c9034_124.png)

![](img/cf7c4c79136b818b283308f5417c9034_125.png)

![](img/cf7c4c79136b818b283308f5417c9034_127.png)

![](img/cf7c4c79136b818b283308f5417c9034_129.png)

**思路**：这是一个博弈论问题，寻找最优混合策略。目标是使期望收益独立于我选择的数字N。
**策略**：以正比于 `1/K` 的概率来猜数字K。即，猜K的概率为 `(1/K) / sum(1/i for i in 1..100)`。
**原因**：在这种策略下，无论我选哪个数字N，你的期望收益都是恒定的。例如，我选N，你猜中的概率是 `(1/N)/sum`，猜中后获得N元，所以这次猜对的贡献期望是 `(1/N)/sum * N = 1/sum`。因此总期望与N无关。

### 2. 最大回撤（Max Drawdown）

![](img/cf7c4c79136b818b283308f5417c9034_131.png)

**问题**：给定一个价格序列，计算其最大回撤。

**思路**：使用动态规划（迭代）的思想，一次遍历解决。
*   维护两个变量：`current_max`（到当前位置为止的历史最高价）和 `max_drawdown`（全局最大回撤）。
*   遍历每个价格`price[i]`：
    *   更新 `current_max = max(current_max, price[i])`。
    *   计算当前回撤 `drawdown = current_max - price[i]`。
    *   更新 `max_drawdown = max(max_drawdown, drawdown)`。

![](img/cf7c4c79136b818b283308f5417c9034_133.png)

![](img/cf7c4c79136b818b283308f5417c9034_135.png)

**核心代码公式**：
```
初始化 current_max = prices[0], max_dd = 0
For i in 1 to n-1:
    current_max = max(current_max, prices[i])
    current_dd = current_max - prices[i]
    max_dd = max(max_dd, current_dd)
返回 max_dd
```
