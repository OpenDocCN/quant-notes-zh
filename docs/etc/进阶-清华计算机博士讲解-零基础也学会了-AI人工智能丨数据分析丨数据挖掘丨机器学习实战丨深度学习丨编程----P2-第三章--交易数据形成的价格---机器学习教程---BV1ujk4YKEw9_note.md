## 概述：为什么要进行回测？🤔

量化交易与主观交易的最大区别在于其可复现性。在将模型投入实盘交易前，我们必须通过历史数据验证其可行性。这个过程就是回测。本节我们将从基础开始，搭建自己的回测工具。

## 第一部分：使用Excel进行简单回测 📈

上一节我们介绍了数据合成，本节我们来看看如何利用这些数据。我们将从一个简单的移动平均线交叉策略开始，在Excel中手动计算其净值曲线。

### 数据准备与移动平均线

首先，我们需要价格序列数据。这里使用的是螺纹钢期货的分钟收盘价数据。交易日从前一晚的夜盘开始计算。

![](img/7af9355585b937fd54d7550f1d29069f_7.png)

![](img/7af9355585b937fd54d7550f1d29069f_8.png)

移动平均线是常用的技术指标。**MA5** 代表最近5根K线的收盘价平均值，**MA20** 代表最近20根K线的收盘价平均值。其计算公式如下：

![](img/7af9355585b937fd54d7550f1d29069f_10.png)

![](img/7af9355585b937fd54d7550f1d29069f_12.png)

`MA(n) = (Price_t + Price_{t-1} + ... + Price_{t-n+1}) / n`

在Excel中，我们可以使用 `AVERAGE` 函数配合偏移引用来计算。

### 构建交易策略

我们的简单策略规则如下：
*   当 **MA5 >= MA20** 时，做多一手。
*   当 **MA5 < MA20** 时，做空一手。

该策略利用了价格的动量效应。

### 计算损益与净值曲线

![](img/7af9355585b937fd54d7550f1d29069f_14.png)

![](img/7af9355585b937fd54d7550f1d29069f_15.png)

以下是计算损益和净值的关键步骤列表：

1.  **计算仓位**：根据策略信号，生成每一分钟的持仓（1为多仓，-1为空仓）。
2.  **计算持仓损益**：持仓损益反映了仅因价格变动而产生的盈亏，不考虑交易动作。公式为：
    `Position PnL = Current Position * (Current Close - Previous Close) * Contract Multiplier`
3.  **计算交易损益**：交易损益反映了因仓位变化（开仓、平仓）而产生的盈亏。公式为：
    `Trade PnL = (Current Position - Previous Position) * (Previous Close - Current Close) * Contract Multiplier`
    *注意：此公式符号设计使得与持仓损益相加后，能得到真实的分钟总损益。*
4.  **计算总损益与净值**：总损益是持仓损益与交易损益之和。累计总损益加上初始资金，就得到了资金曲线。净值则是当前资金除以初始资金。

**核心关系**：`Total PnL = Position PnL + Trade PnL`

通过这个Excel示例，我们验证了该策略在测试期内获得了约15%的回报。但请注意，这只是一个演示，未考虑手续费、滑点等因素，切勿直接用于实盘。

## 第二部分：使用Python实现向量化回测 🐍

虽然Excel直观，但处理大数据量时效率低下且不易扩展。接下来，我们用Python实现一个向量化的回测引擎。

### 向量化计算的优势

向量化操作利用NumPy/Pandas的底层优化，避免使用缓慢的Python循环，能极大提升计算速度，便于批量测试大量因子或参数。

以下是核心类结构的功能列表：

*   **DataHandler**：负责数据加载与预处理，返回一个包含时间、收盘价等信息的DataFrame。
*   **Backtest基类**：定义回测框架。
    *   `__init__`：初始化，载入数据。
    *   `generate_signal` (**抽象方法**)：子类必须实现，用于生成交易信号。
    *   `generate_position`：根据信号生成仓位。可以在此处实现仓位管理逻辑（如固定手数、基于信号强度动态调整等）。

*   **策略子类**：继承Backtest基类，实现具体的`generate_signal`方法。

## 第三部分：迈向事件驱动型回测框架 ⚙️

上一节我们介绍了向量化回测，本节我们来看看更复杂、更贴近实盘的回测架构——事件驱动型回测。

### 向量化回测的局限性

向量化回测虽然高效，但难以模拟复杂的交易逻辑，例如：
*   **止盈止损**：当盈利或亏损达到特定阈值时平仓。
*   **移动止盈**：跟踪最高盈利，在回撤一定幅度时平仓。
*   **价差交易**：同时交易两个相关合约，并考虑各自的盘口和成交情况。
*   **精细成交模拟**：考虑涨跌停、挂单排队、部分成交等情况。

### 事件驱动框架的核心概念

事件驱动框架将整个交易过程抽象为一系列事件，并按时间顺序处理。其核心是**队列**数据结构，遵循**先进先出**原则。

![](img/7af9355585b937fd54d7550f1d29069f_46.png)

![](img/7af9355585b937fd54d7550f1d29069f_48.png)

以下是框架中的核心组件及其职责列表：

![](img/7af9355585b937fd54d7550f1d29069f_50.png)

![](img/7af9355585b937fd54d7550f1d29069f_52.png)

*   **EventQueue**：事件队列，所有事件按发生时间顺序进入队列等待处理。
*   **MarketDataEngine**：市场数据引擎，负责读取历史数据或接收实时数据，并生成`MarketEvent`推入队列。
*   **Strategy**：策略类，从队列中获取`MarketEvent`，根据算法生成`SignalEvent`（交易信号）并推回队列。
*   **Portfolio**：投资组合管理类，从队列中获取`SignalEvent`，结合当前持仓、资金、风控规则，生成`OrderEvent`（订单事件）并推回队列。它也负责处理成交回报，更新持仓和资金。
*   **ExecutionEngine**：交易执行引擎，从队列中获取`OrderEvent`，根据当前市场状态（如盘口）判断能否成交，并生成`FillEvent`（成交事件）推回队列。

### 事件处理流程

![](img/7af9355585b937fd54d7550f1d29069f_65.png)

整个回测由一个主循环驱动：
1.  检查`EventQueue`是否为空。
2.  若非空，则从队列中取出最早的事件。
3.  根据事件的类型，将其分发给对应的组件处理（`MarketDataEngine`, `Strategy`, `Portfolio`, `ExecutionEngine`）。
4.  处理组件可能会产生新的事件并推入`EventQueue`。
5.  重复步骤1，直到所有历史数据处理完毕且队列为空。
