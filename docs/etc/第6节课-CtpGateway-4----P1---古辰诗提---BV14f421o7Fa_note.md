## 🕐 时区设置详解

![](img/845879d628653a4b101625e24610ddf8_9.png)

![](img/845879d628653a4b101625e24610ddf8_11.png)

![](img/845879d628653a4b101625e24610ddf8_13.png)

在vn.py中，处理时间时区主要涉及两个类：`tzlocal` 和 `pytz`。如果你的策略只交易国内品种，可以安全地删除所有时区转换代码。

![](img/845879d628653a4b101625e24610ddf8_15.png)

![](img/845879d628653a4b101625e24610ddf8_17.png)

以下是时区设置的核心方法：

```python
from datetime import datetime
import pytz
from tzlocal import get_localzone

![](img/845879d628653a4b101625e24610ddf8_19.png)

![](img/845879d628653a4b101625e24610ddf8_21.png)

# 方法一：使用pytz（需注意类型）
tz_shanghai = pytz.timezone('Asia/Shanghai')
dt_utc = datetime.now()
dt_with_tz = dt_utc.replace(tzinfo=tz_shanghai)  # 注意：tzinfo参数需要的是tzinfo对象

# 方法二：使用tzlocal获取本地时区
local_tz = get_localzone()
dt_with_local_tz = dt_utc.astimezone(local_tz)
```

关键点在于，`pytz.timezone` 返回的是 `tzinfo` 对象，而 `get_localzone()` 返回的也是可直接使用的时区对象。如果代码报错，通常是类型不匹配导致的。

## 📊 Tick数据转换与价格调整

![](img/845879d628653a4b101625e24610ddf8_27.png)

![](img/845879d628653a4b101625e24610ddf8_29.png)

![](img/845879d628653a4b101625e24610ddf8_31.png)

`onRtnDepthMarketData` 回调函数负责将交易所传来的字典数据转换为vn.py内部的 `TickData` 对象。其中一个关键步骤是价格调整（`adjust_price`）。

![](img/845879d628653a4b101625e24610ddf8_33.png)

![](img/845879d628653a4b101625e24610ddf8_35.png)

`adjust_price` 函数的作用是处理无效价格。交易所对于盘中尚未产生的数据（如收盘价、结算价），有时会发送一个极大值（`sys.float_info.max`）或0作为占位符。该函数将这些无效值转换为0。

![](img/845879d628653a4b101625e24610ddf8_37.png)

![](img/845879d628653a4b101625e24610ddf8_39.png)

![](img/845879d628653a4b101625e24610ddf8_41.png)

```python
import sys

![](img/845879d628653a4b101625e24610ddf8_43.png)

![](img/845879d628653a4b101625e24610ddf8_45.png)

![](img/845879d628653a4b101625e24610ddf8_47.png)

def adjust_price(price: float) -> float:
    """调整价格，将无效的极大值转换为0。"""
    if price == sys.float_info.max:
        price = 0
    return price
```

![](img/845879d628653a4b101625e24610ddf8_49.png)

**核心概念**：`sys.float_info.max` 是Python浮点数能表示的最大值，约为 **1.79769e+308**。在Tick数据中，如果 `close_price`（收盘价）或 `settlement_price`（结算价）等于此值，代表该数据在盘中无效。

![](img/845879d628653a4b101625e24610ddf8_51.png)

![](img/845879d628653a4b101625e24610ddf8_53.png)

![](img/845879d628653a4b101625e24610ddf8_55.png)

不同交易所在数据格式上可能有差异，例如上期所（SHFE）的浮点数可能带有多个尾随零。这是计算机浮点数存储的固有特性（浮点误差），在大多数情况下不影响使用，但需要知晓。

### 1. 合约信息 (`onRspQryInstrument`)

![](img/845879d628653a4b101625e24610ddf8_73.png)

![](img/845879d628653a4b101625e24610ddf8_75.png)

![](img/845879d628653a4b101625e24610ddf8_77.png)

这是连接交易服务器后首先（或同步）处理的数据。`ContractData` 包含了交易合约的所有基础信息，如合约乘数（`size`）、最小变动价位（`pricetick`）等，这些是后续计算委托价格、持仓市值的基础。

![](img/845879d628653a4b101625e24610ddf8_79.png)

![](img/845879d628653a4b101625e24610ddf8_81.png)

![](img/845879d628653a4b101625e24610ddf8_83.png)

![](img/845879d628653a4b101625e24610ddf8_85.png)

处理逻辑如下：
*   服务器返回合约信息字典，将其转换为 `ContractData` 对象。
*   将合约信息存入全局字典 `symbol_contract_map`，供其他函数查询。
*   如果是最后一批合约数据（`isLast=True`），则标记合约查询完成，并开始处理之前缓存起来的委托和成交数据。

![](img/845879d628653a4b101625e24610ddf8_87.png)

![](img/845879d628653a4b101625e24610ddf8_89.png)

**自定义建议**：如果只交易期货，可以删除代码中所有与期权（`Product.OPTION`）相关的部分，使代码更简洁。

### 2. 委托回报 (`onRtnOrder`)

![](img/845879d628653a4b101625e24610ddf8_91.png)

![](img/845879d628653a4b101625e24610ddf8_93.png)

当委托状态发生变化（如报单、部分成交、全部成交、撤单、拒单）时，此函数被调用。

![](img/845879d628653a4b101625e24610ddf8_95.png)

![](img/845879d628653a4b101625e24610ddf8_97.png)

**关键点：本地委托ID与系统委托ID的映射**
*   策略层发出的委托会生成一个本地唯一的 `orderid`（由 `frontid`, `sessionid`, `order_ref` 拼接而成）。
*   委托到达交易所后，交易所会分配一个唯一的 `order_sysid`。
*   `order_sysid_map` 这个字典用于建立 `order_sysid` 和本地 `orderid` 的映射关系。这样，当交易所返回委托状态或成交回报时，vn.py才能找到对应的本地委托对象并更新其状态。

![](img/845879d628653a4b101625e24610ddf8_99.png)

### 3. 成交回报 (`onRtnTrade`)

![](img/845879d628653a4b101625e24610ddf8_101.png)

![](img/845879d628653a4b101625e24610ddf8_103.png)

![](img/845879d628653a4b101625e24610ddf8_105.png)

当委托产生成交时，此函数被调用。它会通过 `order_sysid_map` 找到对应的本地委托，并更新该委托的已成交量（`traded`）等状态。

![](img/845879d628653a4b101625e24610ddf8_107.png)

![](img/845879d628653a4b101625e24610ddf8_109.png)

### 4. 持仓查询 (`onRspQryInvestorPosition`)

用户主动查询或定时查询持仓时，服务器返回持仓信息。处理持仓的逻辑相对复杂，因为需要处理以下情况：
*   **多策略共用一个合约**：多个策略对同一合约的持仓方向可能相反，实际持仓是净头寸。
*   **上期所（SHFE）的特殊性**：平仓时需要明确指定是平“今仓”还是平“昨仓”。vn.py的“转换器”（`OffsetConverter`）模块会处理此逻辑，可能将一个平仓委托拆分成平今和平昨两个委托。
*   **冻结持仓**：已发出但尚未成交的平仓委托，会冻结对应的持仓数量，避免重复发单。

![](img/845879d628653a4b101625e24610ddf8_111.png)

![](img/845879d628653a4b101625e24610ddf8_113.png)

持仓查询回报会计算持仓成本、浮动盈亏，并更新到主引擎（`main_engine`）中。

### 5. 资金查询 (`onRspQryTradingAccount`)

查询账户资金信息，包括余额、可用资金、冻结资金、持仓保证金、手续费等。这是一个简单的数据赋值过程。

## 💡 实践建议与总结

![](img/845879d628653a4b101625e24610ddf8_115.png)

![](img/845879d628653a4b101625e24610ddf8_117.png)

本节课我们一起深入学习了CtpGateway交易接口的各个组成部分。为了彻底掌握并构建稳定的交易系统，建议进行以下实践：

1.  **数据输出分析**：运行网关，将所有收到的原始 `TickData`、`ContractData`、`OrderData`、`TradeData` 打印出来，直观了解其结构和内容差异。
2.  **理解数据流**：画出一个简单的数据流图，从策略发单到交易所回报，理清 `orderid`、`order_sysid`、`order_sysid_map` 在整个流程中的作用。
3.  **模拟测试**：在实盘前，务必使用历史Tick数据或模拟账户进行充分测试。验证K线合成、策略信号、委托触发与成交回报的整个闭环是否准确无误，特别注意开盘集合竞价、无效Tick过滤等边界情况。
4.  **大胆修改**：基于你对代码的理解，如果某些功能（如期权支持、复杂的时区转换）用不到，可以尝试删除或简化相关代码。在备份的前提下，动手修改是学习的最佳途径。
