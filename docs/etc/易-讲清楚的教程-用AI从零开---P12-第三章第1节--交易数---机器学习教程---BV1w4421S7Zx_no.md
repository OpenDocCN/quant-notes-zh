## 第一部分：Excel回测入门 📈

上一节我们介绍了量化交易的基本概念，本节中我们来看看如何用最基础的工具——Excel，来实现一个简单的策略回测。

### 数据来源与处理
首先，我们需要交易数据。数据来源于第一课作业中合成的一分钟K线（Bar）。数据包含`datetime`和`close`价格两列。需要注意的是，`datetime`为`21:00`代表从`21:00`到`21:01`这一分钟内的所有时间点。交易日通常从前一晚的夜盘开始计算。

下图展示了螺纹钢在2018年11月30日至12月7日期间的分钟收盘价序列。
![](img/f467f2de0fa9ab84562add150e5b7e42_1.png)

### 移动平均线策略
我们的目标是构建一个简单的交易策略。首先，引入一个核心概念：**移动平均线**。

**移动平均线**的计算公式为：
`MA(n) = (P1 + P2 + ... + Pn) / n`
其中，`P`代表价格，`n`代表周期。例如，`MA5`代表最近5根Bar的收盘价平均值。

![](img/f467f2de0fa9ab84562add150e5b7e42_7.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_9.png)

在Excel中，我们可以用公式填充的方式计算`MA5`和`MA20`。注意，前4根Bar没有`MA5`值，因为数据长度不足。

![](img/f467f2de0fa9ab84562add150e5b7e42_11.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_13.png)

### 策略逻辑与仓位
我们定义一个简单的双均线策略：
*   当 `MA5 >= MA20` 时，做多一手。
*   当 `MA5 < MA20` 时，做空一手。

这个策略利用了市场的动量效应。我们的任务就是将这个交易逻辑转化为一条资金净值曲线。

### 损益计算：Position P&L 与 Trade P&L
这是回测中的关键概念。我们需要计算每一分钟的损益。

**Position P&L（持仓损益）**：在持仓不变的情况下，由于价格变动产生的损益。
计算公式为：
`Position P&L = 当前持仓 * (当前Bar收盘价 - 上一Bar收盘价) * 合约乘数`

**Trade P&L（交易损益）**：由于仓位变化（交易）产生的损益。
计算公式为：
`Trade P&L = (当前持仓 - 上一Bar持仓) * (-上一Bar收盘价) * 合约乘数`
*注意：价格变化与仓位变化的方向符号是相反的。*

**总损益**是两者之和：
`Total P&L = Position P&L + Trade P&L`

为什么要区分两者？这是为了准确计算每一笔交易的真实盈亏。例如，如果在本Bar结束时以`3230`的价格开多一手，但本Bar的收盘价也是`3230`，那么这笔开仓交易本身没有产生`Trade P&L`，但持仓会承受之后价格波动的`Position P&L`。区分二者有助于未来测试不同的挂单成交价对策略的影响。

![](img/f467f2de0fa9ab84562add150e5b7e42_15.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_17.png)

### 构建净值曲线
以下是构建净值曲线的步骤列表：
1.  **计算累计损益**：将每一Bar的`Total P&L`累加，得到累计损益序列。
2.  **计算资金曲线**：设定初始资金（例如1万元），然后加上累计损益，得到资金曲线。
3.  **计算净值序列**：有两种等价方法。
    *   方法一：`当前净值 = 当前资金 / 初始资金`
    *   方法二：先计算每一Bar的收益率 `收益率 = (当前资金 - 上Bar资金) / 上Bar资金`，然后通过 `当前净值 = 上Bar净值 * (1 + 当前收益率)` 递推计算。

最终，我们得到了策略的净值曲线。在示例的几天数据中，1万元本金变成了约10530元，收益显著。但这只是一个简化示例，未考虑手续费、滑点等因素，切勿直接用于实盘。

通过这个简单的Excel回测，我们理解了从数据、信号、仓位到净值曲线的完整链条。所有回测本质上都是在做这件事：在任意时间点给定一个仓位，就能计算出最终的净值曲线。

## 第二部分：Python向量化回测引擎 🐍

上一节我们使用Excel手动完成了回测，本节中我们来看看如何用Python自动化这个过程，提高效率。

### 作业：复现Excel回测逻辑
首先，我们需要用Python复现Excel中的所有计算。这是一个向量化操作，意味着应避免使用`for`循环，而是利用`Pandas`等库的数组运算能力。

需要实现的步骤包括：
1.  计算`MA5`和`MA20`。
2.  根据均线关系生成交易信号。
3.  将信号转化为仓位（例如，1代表多，-1代表空）。
4.  计算`Position P&L`、`Trade P&L`和`Total P&L`。
5.  计算累计损益、资金和净值曲线。

完成代码后，应与Excel计算结果完全一致。

### 构建可扩展的回测类
为了测试不同指标和参数，我们需要一个更通用的回测框架。这里引入**面向对象编程**的概念，我们将构建一个回测基类。

![](img/f467f2de0fa9ab84562add150e5b7e42_28.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_30.png)

以下是核心类的设计列表：
*   **DataHandler**：负责数据输入，无论数据来自CSV文件还是数据库，它都返回一个统一的`DataFrame`。
*   **Backtest基类**：所有策略回测类的父类。
    *   初始化时加载数据。
    *   包含一个必须由子类实现的抽象方法 `generate_signal()`，用于生成交易信号。
    *   包含一个方法 `generate_position()`，负责将信号转化为具体的仓位。这允许我们灵活地管理仓位，例如设置空仓、加仓或根据信号强度调整仓位。
*   **向量化回测子类**：继承自`Backtest`基类，并实现`calculate_performance`方法，其中封装了之前所有的损益和净值计算逻辑。

![](img/f467f2de0fa9ab84562add150e5b7e42_32.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_33.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_34.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_35.png)

这样设计的好处是，我们将策略逻辑（`generate_signal`）和绩效计算逻辑解耦。未来要测试新策略时，只需继承基类并重写`generate_signal`方法即可，无需改动核心的回测计算部分。

### 加入更真实的交易假设
在基础版本上，我们可以加入更贴近现实的假设：
*   **成交价**：不以收盘价成交，而是以下一Bar的开盘价成交，并用`Ask`价买入、`Bid`价卖出，以模拟交易中的买卖价差。
*   **涨跌停过滤**：过滤掉处于涨跌停状态的Bar，因为在这些情况下订单可能无法成交。
*   **手续费**：在`Trade P&L`中扣除每笔交易的手续费。

## 第三部分：事件驱动型回测架构 ⚙️

上一节我们构建了快速的向量化回测，但对于复杂策略（如止盈止损、多合约价差），它可能不够灵活。本节中我们来看看更强大、更贴近实盘的事件驱动型回测架构。

### 为什么需要事件驱动？
向量化回测假设所有交易都在Bar结束时瞬时完成，这与现实不符。实盘中：
*   订单可能无法立即成交。
*   需要处理止盈、止损、移动止盈止损等动态风控逻辑。
*   需要同时处理多个合约的交易和资金分配。
事件驱动架构通过模拟真实世界中事件发生的先后顺序来处理这些复杂情况。

### 核心：队列与事件循环
该架构的核心是**队列**数据结构，其特点是**先进先出**。我们将交易过程中发生的所有事情抽象为**事件**，并按发生顺序放入队列中。

以下是主要的四类事件：
1.  **MarketEvent**：新的市场数据到来。
2.  **SignalEvent**：策略根据市场数据产生交易信号。
3.  **OrderEvent**：投资组合管理器根据信号产生订单。
4.  **FillEvent**：交易所（撮合引擎）报告订单成交。

### 系统组件与工作流程
整个系统由四个核心组件构成，它们通过事件队列进行通信。

![](img/f467f2de0fa9ab84562add150e5b7e42_47.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_49.png)

以下是各组件职责与流程的简述：
1.  **市场数据引擎**：发布`MarketEvent`到队列。
2.  **策略类**：从队列获取`MarketEvent`，计算并发布`SignalEvent`到队列。
3.  **投资组合类**：从队列获取`SignalEvent`，结合当前持仓、资金、风控规则，发布`OrderEvent`到队列。
4.  **执行引擎**：从队列获取`OrderEvent`，模拟交易所撮合。若成交，则发布`FillEvent`到队列。`FillEvent`会再次被**投资组合类**获取，用于更新持仓、资金和计算实时损益。

![](img/f467f2de0fa9ab84562add150e5b7e42_51.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_53.png)

整个回测过程就是一个持续的**事件循环**：只要队列不为空，就取出最前面的事件，根据其类型分发给相应的组件处理，处理过程中可能产生新的事件并放入队列。当所有历史市场数据消费完毕且队列为空时，回测结束。

![](img/f467f2de0fa9ab84562add150e5b7e42_55.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_57.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_59.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_61.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_63.png)

![](img/f467f2de0fa9ab84562add150e5b7e42_65.png)

这种架构高度模块化，每个组件只关心自己的输入和输出事件，彼此解耦。这使得策略研发、风险管理和订单执行可以独立开发和优化，非常适合构建复杂的交易系统。
