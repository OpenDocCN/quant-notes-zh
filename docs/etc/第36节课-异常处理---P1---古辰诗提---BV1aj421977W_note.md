*   **语法错误**：这是最基础的错误，通常由代码书写不符合Python语法规则引起。例如，在Python 3中使用Python 2的`print`语法。
    ```python
    print “Hello World”  # 错误的语法
    ```
*   **断言错误**：当`assert`语句的条件为`False`时触发。常用于调试阶段检查程序状态。
    ```python
    def number_count(number):
        assert number > 0  # 如果number不大于0，则触发AssertionError
        while number > 0:
            print(number)
            number -= 1
    ```
*   **索引错误**：当尝试访问列表、元组等序列中不存在的索引时发生。
    ```python
    my_list = []
    print(my_list[1])  # 列表为空，索引1不存在，触发IndexError
    ```
*   **名称错误**：当尝试访问一个未定义的变量时发生。
    ```python
    print(a)  # 变量a未定义，触发NameError
    ```
*   **属性错误**：当尝试访问对象不存在的属性时发生。
    ```python
    class Person:
        pass
    p = Person()
    print(p.name)  # Person类没有name属性，触发AttributeError
    ```
*   **键错误**：当尝试访问字典中不存在的键时发生。
    ```python
    my_dict = {}
    print(my_dict[‘key’])  # 字典中没有‘key’这个键，触发KeyError
    ```
*   **类型错误**：当操作或函数应用于不适当类型的对象时发生。
    ```python
    result = ‘5’ + 6  # 字符串和整数不能直接相加，触发TypeError
    ```
*   **零除错误**：当除法或取模运算的除数为零时发生。
    ```python
    result = 1 / 0  # 除数为零，触发ZeroDivisionError
    ```

异常处理的基本语法结构如下：
```python
try:
    # 尝试执行的代码块
    可能出错的代码
except 异常类型 as 变量名:
    # 如果发生指定类型的异常，则执行这里的代码
    处理异常的代码
else:
    # 如果没有发生异常，则执行这里的代码
    无异常时执行的代码
finally:
    # 无论是否发生异常，最终都会执行的代码
    始终会执行的代码
```

```python
def divide_numbers():
    try:
        a = int(input(“请输入除数: “))
        b = int(input(“请输入被除数: “))
        c = b / a
        print(f“结果是: {c}“)
    except ZeroDivisionError as e:
        print(f“错误：除数为零。详细信息: {e}“)
    except ValueError as e:
        print(f“错误：输入类型有误，请输入数字。详细信息: {e}“)
    except Exception as e:
        print(f“出现未知错误: {e}“)
    else:
        print(“计算成功完成。”)
    finally:
        print(“除法运算执行完毕。”)
```

*   **`try`块**：包含可能引发异常的代码。
*   **`except`块**：用于捕获和处理特定的异常。可以指定多个`except`块来处理不同类型的异常。使用`as e`可以将异常对象赋值给变量`e`，以便获取错误信息。
*   **`else`块**：当`try`块中的代码没有引发任何异常时执行。
*   **`finally`块**：无论是否发生异常，最终都会执行的代码块，常用于执行清理操作（如关闭文件）。**注意：在`finally`块中应避免使用`return`语句**。

有时我们不仅需要捕获异常，还需要获取其完整的调用栈信息，以便于调试。Python的`traceback`模块提供了这个功能。

![](img/ae8f25357e74c7b6541738d04804fcc5_155.png)

![](img/ae8f25357e74c7b6541738d04804fcc5_157.png)

![](img/ae8f25357e74c7b6541738d04804fcc5_159.png)

![](img/ae8f25357e74c7b6541738d04804fcc5_161.png)

以下是两种常用的方法：

![](img/ae8f25357e74c7b6541738d04804fcc5_163.png)

![](img/ae8f25357e74c7b6541738d04804fcc5_164.png)

![](img/ae8f25357e74c7b6541738d04804fcc5_166.png)

![](img/ae8f25357e74c7b6541738d04804fcc5_168.png)

*   **`traceback.print_exc()`**：直接将异常的追踪信息打印到控制台。可以通过`limit`参数限制显示的追踪层级，通过`file`参数指定输出到文件。
    ```python
    import traceback
    try:
        # 可能出错的代码
        1 / 0
    except Exception:
        traceback.print_exc(limit=1)  # 只打印最近一层的错误信息
    ```
*   **`traceback.format_exc()`**：将异常的追踪信息格式化为字符串，方便存储或进一步处理。
    ```python
    import traceback
    try:
        # 可能出错的代码
        1 / 0
    except Exception:
        error_msg = traceback.format_exc()
        print(“捕获到错误:”)
        print(error_msg)  # 将错误信息字符串打印出来
        # 也可以写入日志文件：with open(‘error.log’, ‘a’) as f: f.write(error_msg)
    ```
在实际项目（例如量化交易平台）中，`format_exc()`常用于将错误信息记录到日志文件中，便于后续排查问题，而不会让错误信息淹没在控制台输出中。

除了处理系统抛出的异常，我们也可以使用`raise`语句主动抛出异常。这通常用于强制要求某些条件必须满足，或者在检测到非法状态时提前终止程序流程。

`raise`语句有三种常见用法：

![](img/ae8f25357e74c7b6541738d04804fcc5_199.png)

![](img/ae8f25357e74c7b6541738d04804fcc5_201.png)

1.  **`raise`**：单独使用，重新引发当前上下文中最后捕获的异常。如果当前没有异常，则引发一个`RuntimeError`。
    ```python
    try:
        1 / 0
    except ZeroDivisionError:
        print(“捕获到零除错误”)
        raise  # 重新抛出这个ZeroDivisionError
    ```
2.  **`raise 异常类型`**：抛出一个指定类型的异常。
    ```python
    def check_positive(number):
        if number <= 0:
            raise ValueError  # 抛出一个ValueError
        return number
    ```
3.  **`raise 异常类型(提示信息)`**：抛出一个指定类型的异常，并附带自定义的错误描述信息。
    ```python
    def check_positive(number):
        if number <= 0:
            raise ValueError(“输入的数字必须为正数”)  # 抛出带提示信息的ValueError
        return number
    ```
主动抛出异常可以让我们的代码逻辑更清晰，错误提示更友好。

关于异常处理，有几点最佳实践建议：
1.  **精准捕获**：尽量捕获具体的异常类型（如`ValueError`, `KeyError`），而不是盲目使用通用的`except Exception`。这有助于避免隐藏潜在的程序逻辑错误。
2.  **记录异常**：在生产环境中，使用`logging`模块或`traceback.format_exc()`将异常信息记录到日志文件中，而不是仅仅打印到控制台。
3.  **保持简洁**：异常处理不应滥用。不要用`try...except`包裹大段正常逻辑，它只应用于处理那些你预期可能发生且你知道如何处理的“异常”情况。对于普通的程序错误，应优先考虑修改代码逻辑来避免。
4.  **清理资源**：将释放资源（如关闭文件、网络连接）的代码放在`finally`块中，确保其无论如何都会被执行。
