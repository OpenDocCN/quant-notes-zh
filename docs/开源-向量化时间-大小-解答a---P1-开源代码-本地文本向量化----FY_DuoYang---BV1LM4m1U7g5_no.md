# 开源向量化知识库教程：P1：CPU环境下文本向量化的时间与资源占用分析 🧠

在本节课中，我们将学习如何在本地的CPU环境下进行文本向量化，这是构建本地AI知识库（如RAG应用）的关键步骤。我们将通过一个开源项目，具体分析处理不同大小文本时所需的时间以及生成的向量文件所占用的硬盘空间。理解这些数据有助于你规划自己的知识库项目，并解释为何有时AI不会基于你上传的知识进行回答。

## 概述：向量化与本地AI知识库

许多本地AI知识库项目都需要一个必要的步骤：**文本向量化**。通过把文本转化为向量，可以实现**检索增强生成（RAG）**，从而提升AI回答的质量和相关性。由于本地程序大多运行在资源有限的小型机器上，且以CPU计算为主，因此了解CPU下的向量化性能至关重要。

本节实验旨在测试在CPU环境下，对不同量的文本进行向量化所需的时间，以及生成的向量文件所占用的硬盘资源大小。

## AI不按知识库回答的原因与系统负载

![](img/5d03767ad8391a78b9d15e240f1e320f_2.png)

很多人在使用AI知识库时会发现，自己的提问没有经过知识库的内容进行回答。这通常是由于**本地向量化过程尚未完成**导致的。在向量化进行时，CPU会一直处于高负载状态，此时打开其他程序可能会感到卡顿。

## 核心参数：文本分割大小

简单讲一下代码中的关键参数。其中一个核心影响向量化时间和资源占用的参数是**文本分割大小**。

以下是该参数的影响逻辑：
*   **分割文本越小**：生成的文本片段数量就越多。
*   **片段数量越多**：整体向量化所需的时间就越长，最终占用的硬盘资源也越大。

当然，如果你直接向量化一个完整的文档（例如1个EMB文件），速度会很快，但这样的大片段在后续检索中意义不大，因为无法精确定位到具体信息。

## 总结

![](img/5d03767ad8391a78b9d15e240f1e320f_4.png)

本节课中，我们一起学习了在CPU环境下进行本地文本向量化的实践。我们了解到：
1.  向量化是构建RAG式AI知识库的核心前置步骤。
2.  使用本地模型（如M3E）可以避免网络延迟。
3.  **文本分割大小**是一个关键参数，它需要在**处理速度**、**硬盘占用**和**检索精度**之间取得平衡。
4.  向量化过程会显著占用CPU资源，可能导致系统卡顿，这也是知识库未就绪时AI无法基于知识回答的原因之一。
5.  成功向量化后，即可通过向量搜索来检索相关知识，从而增强AI的回答能力。

通过本教程的分析，你可以更好地规划和优化自己的本地AI知识库项目。
